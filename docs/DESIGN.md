## Problem Statement

Build a distributed-systems learning project that produces a Go control plane for
Envoy. The control plane discovers backends from Kubernetes, pushes xDS config to
Envoy fleets, and runs a short feedback loop: it scrapes live latency telemetry
from each Envoy and uses it to drive routing weights in near real time.

This is a learning project, not a startup or production system. The goal is to
deeply understand control-plane / data-plane separation, sliding-window telemetry,
EWMA-based decisioning, and the partial-failure semantics of pushing live config
to a fleet — by building it, not by reading about it.

## What Makes This Cool

The "whoa" is a live demo on a laptop. Inject 200ms of latency on one backend
with `tc qdisc add dev eth0 root netem delay 200ms` and watch traffic drain from
that pod within 3–5 seconds, with no client-visible errors. Then flip a flag and
watch a 10/90 canary smoothly become 50/50 over 30 seconds. The control plane
*reacts* — most production setups push static weights and rely on Envoy's local
outlier detection. This one uses the data plane's own telemetry to make routing
decisions globally.

Most "build a load balancer" projects rewrite a TCP forwarder. This one skips
the byte-pushing entirely and goes straight at the harder, more educational
problem: how do you make a fleet of dataplanes do something coherent based on
live signal?

## Constraints

- Solo developer, time-boxed weekends (target V1: 2–3 weekends)
- Runs on a laptop with `kind` or `k3d` for the V1 demo
- Go for the control plane (one process for V1, no clustering yet)
- Envoy as the data plane (do not write a data plane)
- One discovery source for V1: Kubernetes
- No production deployment; demo-quality is the bar

## Premises

The five premises agreed during office hours:

1. **Use `github.com/envoyproxy/go-control-plane`, not hand-rolled xDS.**
   The SDK gives you snapshot cache, ADS multiplexing, version tracking. Writing
   xDS from scratch is ~6 weeks of yak-shaving on protobuf and gRPC streaming
   that has nothing to do with distributed systems. Spend that time on the
   feedback loop.

2. **Demo-ready, not production-ready, for V1.**
   "Distributed HA control plane, millions of connections" is V3+. V1 is one
   control plane process on a laptop, kind/k3d cluster, three demo services,
   two Envoys, one beautiful feedback-loop demo. The HA learning happens in
   V2 — but you need V1 to be real first to know what V2 should do.

3. **One discovery source for V1: Kubernetes.**
   Multi-source (k8s + Consul + etcd + DNS + static) multiplies adapter code
   without multiplying learning. Add others only if a real reason emerges.

4. **The feedback loop is the project, not the load balancer.**
   Static config push (watch k8s, push EDS) is ~500 lines and uninteresting.
   The novel and educational part is: scrape Envoy `/stats`, build sliding-
   window p50/p99 per upstream, run EWMA + power-of-2-choices, push live-
   weighted EDS updates. This is where the distributed-systems learning lives:
   windowing, dampening, oscillation prevention, partial failure semantics.

5. **"Millions of connections" is Envoy's job, not yours.**
   Envoy already handles that. The control plane talks to ~tens of Envoys,
   each emitting ~KB/sec of stats. Tiny system in absolute terms. If you
   find yourself optimizing the control plane for raw throughput, you've
   drifted off-mission.

## Approaches Considered

### Approach A — MVP Failover Demo
Hardcoded service list in YAML. Active health checks from control plane, push
EDS updates on health change. Demo: kill a pod, see traffic move in <500ms.
**Effort:** 1 weekend. **Tradeoff:** ships fast but doesn't include the
feedback loop, which is the whole point.

### Approach B — The Feedback-Loop Project (CHOSEN)
k8s informer watches Services/Endpoints. xDS server pushes CDS/EDS/RDS/LDS via
go-control-plane. Stat scraper polls Envoy `/stats/prometheus` at 1Hz. Per-
upstream EWMA of p99 latency over a sliding window. Power-of-2-choices weighting.
EDS updates pushed when weights drift past threshold. Small CLI: `lb status`,
`lb shift v1=10 v2=90`. **Effort:** 2–3 weekends.

### Approach C — Visual Showcase
Approach B plus a live single-page web UI: force-directed service graph, edges
weighted by traffic and colored by latency, drag-to-rebalance.
**Effort:** 4–5 weekends. **Risk:** UI work eats systems-learning time.

## Recommended Approach

**Approach B**, with the following V1 scope:

**In scope for V1:**
- One control plane process (no clustering)
- Kubernetes informer watching `Service` and `Endpoints` (or `EndpointSlices`)
- xDS server using `go-control-plane`'s `SnapshotCache` (start with ADS, not separate streams)
- Active health checks via Envoy's own outlier detection (don't reimplement)
- Stat scraper: HTTP poll of Envoy admin `/stats/prometheus` at 1Hz
- Sliding-window p99 latency per upstream cluster (60s window, ~60 buckets)
- EWMA decay (start with α=0.3, expect to tune)
- Weight assignment: power-of-2-choices over the EWMA-weighted set
- EDS push trigger: weight delta exceeds threshold (e.g., 5%) OR every 10s, whichever first
- CLI: `lb status` (table view of clusters, weights, observed p99); `lb shift <cluster>=<weight>` for manual canary

**Out of scope for V1 (explicit defer list):**
- Multi-cluster federation
- Distributed control plane (multiple control-plane replicas, gossip, sharding)
- mTLS, SPIFFE/SPIRE
- HTTP/3 / QUIC config (Envoy supports it; you can flip the listener config but the demo doesn't need it)
- Web UI
- Multi-source discovery (Consul, etcd, DNS, static files)
- Production hardening (graceful restart, persistent state, leader election)

**Architecture sketch:**

```
┌──────────────────┐         k8s API
│  k8s informer    │◄────── (Services / Endpoints)
└────────┬─────────┘
         │
         ▼
┌──────────────────┐         ┌──────────────────────┐
│ Service registry │ ◄────── │  Stat scraper (1Hz)  │ ◄── /stats/prometheus
└────────┬─────────┘         │   per Envoy          │
         │                   │                      │
         │                   │ EWMA windows,        │
         │                   │ p99 per upstream     │
         │                   └──────────┬───────────┘
         │                              │
         ▼                              ▼
       ┌──────────────────────────────────┐
       │  Weight engine (P2C + EWMA)      │
       │  Decides EDS endpoint weights    │
       └──────────────────┬───────────────┘
                          ▼
       ┌──────────────────────────────────┐
       │  go-control-plane SnapshotCache  │
       │  (CDS / EDS / RDS / LDS)         │
       └──────────────────┬───────────────┘
                          ▼ xDS (gRPC, ADS)
                     ┌─────────┐
                     │ Envoy(s)│
                     └─────────┘
```

## Open Questions

These are real questions you'll have to answer in the build, listed so you don't
forget them:

1. **EndpointSlices vs Endpoints.** Endpoints is the legacy API; EndpointSlices
   scale better and are the future. For V1 with a few services either works.
   Pick one, note the tradeoff in code.
2. **EWMA α tuning.** Start at 0.3. Expect oscillation at first. Build a knob
   in the CLI to retune live so you can experiment.
3. **Cold start: what's the weight of a brand-new endpoint with no telemetry?**
   Equal-weighted (most charitable) or zero-weighted (cautious)? This is a
   real distributed systems question; both have failure modes.
4. **Push debouncing.** A noisy backend can cause weight thrash. Probably need
   a min-time-between-pushes for any single upstream cluster.
5. **What latency does "p99" mean?** Envoy emits histogram buckets — pick a
   reasonable bucket boundary and document it. Don't try to compute "true" p99
   from buckets.
6. **xDS variant.** Use ADS (Aggregated Discovery Service) — single stream,
   easier ordering. Don't start with separate CDS/EDS/RDS/LDS streams.

## Success Criteria

V1 is "done" when, on a laptop, you can:

1. Bring up `kind` with three demo HTTP services (`v1`, `v2`, `v3`) behind one Envoy
2. Send sustained load (e.g., `vegeta` at 200 RPS)
3. Run `tc qdisc add dev eth0 root netem delay 200ms` on `v1`'s pod
4. Watch the control plane CLI show weights shifting away from `v1` within 5 seconds
5. Watch p99 client latency stay flat (no client-visible errors)
6. Run `lb shift v1=0 v2=50 v3=50` and watch traffic move smoothly without dropped requests
7. Tear down `v2` (`kubectl delete pod`) and watch traffic redistribute across `v1` and `v3`

That's the demo. If all seven steps work cleanly, V1 ships.

## Next Steps

In order. Don't skip ahead.

1. **Read three things first** (~3 hours, before writing any code):
   - The Envoy xDS REST/gRPC API overview: `https://www.envoyproxy.io/docs/envoy/latest/api-docs/xds_protocol`
   - `go-control-plane`'s `internal/example` (it ships a working example server)
   - One Istio Pilot blog post on push semantics — pick any from the Istio blog

2. **Stand up scaffolding** (~1 evening):
   - `kind` cluster with three demo HTTP services and one Envoy with a static
     bootstrap that points at your control plane (`localhost:18000`)
   - Skeleton Go binary with `cobra` for `lb status` / `lb shift` / `lb run`
   - Vendor `go-control-plane`, write the simplest possible static snapshot
     and confirm Envoy receives it and routes traffic

3. **Wire k8s discovery → static EDS push** (~1 evening):
   - `client-go` informer on Services / Endpoints
   - On change, generate an updated snapshot, set in SnapshotCache
   - Confirm Envoy picks up endpoint changes when you scale a deployment

4. **Build the stat scraper + EWMA pipeline** (~1 weekend):
   - HTTP client that polls each Envoy's `/stats/prometheus`
   - Parser that pulls `cluster.<name>.upstream_rq_time` histograms
   - Per-cluster sliding window, EWMA, p99 estimate
   - Log it. Don't act on it yet. Just verify the numbers move when you induce latency.

5. **Wire the weight engine** (~1 weekend):
   - P2C over EWMA-weighted set, with min-weight floor
   - Debounced push trigger (5% delta or 10s, whichever first)
   - Run the demo (success criteria #1–7). Expect oscillation. Tune α.

6. **Polish the CLI and record the demo.** Push to GitHub. Show someone.

## The Assignment

**Before you write any code:** spin up a `kind` cluster and a single Envoy
pod, with a static `bootstrap.yaml` that points at your (not-yet-existing)
control plane on `localhost:18000`. Write the bootstrap config by hand, not
generated. Read every line of it. Understand what `node.id`, `node.cluster`,
`xds_cluster`, and the `dynamic_resources` block actually do.

This is one evening of work. It costs almost no code. But it forces you to
internalize the xDS handshake before you write a single Go file. Without this
step, every confusing thing about go-control-plane will be doubly confusing
because you won't have a mental model for what Envoy is asking for.

Once you've done that, post the bootstrap.yaml in a gist or commit it to the
repo. That's the artifact that proves you're ready to start step 2.
